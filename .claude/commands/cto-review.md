# 交叉审核

以下改动涉及关键安全/架构领域，需要独立交叉审核。
请以完全独立的视角审核，不假设前一位审核者的判断是正确的。

## 审核维度

1. **安全合规性**：是否存在注入、泄露、权限绕过等风险
2. **架构一致性**：是否符合项目架构原则，是否引入不必要的耦合
3. **边界条件完整性**：是否覆盖了所有异常路径和边界情况
4. **测试充分性**：N+1 场景、并发场景、回滚场景是否有对应测试
5. **性能影响**：是否引入性能退化，是否有可优化的热路径
6. **产品一致性**：改动是否符合产品愿景和当前迭代目标

## 输出格式

每个发现标注严重程度：
- 🔴 Critical — 必须修复，存在安全或数据风险
- 🟠 Major — 强烈建议修复，影响架构或可维护性
- 🟡 Minor — 可改进，不阻塞
- 🔵 Innovation — 建议采用的更优方案

每条含：具体文件 + 行号/函数 + 问题描述 + 建议修复方案

## 审核方式

**方式一（本地交叉）：** 用 Opus 做第一轮 → sub-agent 做第二轮
**方式二（跨平台）：** 生成两轮委派指令分发给不同平台 Agent
